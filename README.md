# VPN Telegram Bot + WireGuard

Self-hosted VPN-сервис: Telegram-бот на aiogram 3 управляет peer'ами
WireGuard в рантайме, **не создавая файл `wg0.conf`**. Метаданные peer'ов
(включая только публичный ключ) хранятся в SQLite. Приватный ключ клиента
генерируется по запросу, отдаётся пользователю внутри `.conf` и сразу
забывается — на сервере он не сохраняется.

## Архитектура

- Контейнер `wireguard`: Alpine + `wireguard-tools` + `iptables`. Entrypoint
  поднимает интерфейс `wg0` командами `ip link` / `wg set` / `iptables`, то
  есть интерфейс полностью настраивается без конфиг-файла. Peer'ы
  добавляются и удаляются исключительно ботом через `docker exec`.
- Контейнер `bot`: Python 3.12 + aiogram 3. Общается с соседним контейнером
  WireGuard через смонтированный Docker-сокет.
- SQLite лежит в именованном Docker-volume `sqlite_data`, примонтированном
  в контейнер бота.
- Ключи сервера лежат во втором volume `wg_keys`, общем для двух контейнеров
  (бот монтирует его read-only, чтобы читать серверный публичный ключ).

```
Telegram-пользователь
         |
         v
+-----------+       docker exec wg set ...        +---------------+
|   bot     | ---------------------------------->  |  wireguard    |
|           |                                      |  wg0 (runtime)|
| SQLite    | <--- shared volume (pubkey) -------  |               |
+-----------+                                      +---------------+
```

## Аллокация IP

Сеть по умолчанию — `10.0.0.0/22` (1024 адреса, ~1021 под клиентов).
Серверу выделяется `10.0.0.1`. IP-адреса клиентов выдаются обходом
`IPv4Network.hosts()`, а это — 32-битный инкремент: после `10.0.0.255`
следующим адресом идёт `10.0.1.0`, как и требовалось.

Жёсткий лимит `WG_MAX_PEERS` (по умолчанию `1000`) защищает от исчерпания
пула.

## Первый запуск

1. Установи Docker и Docker Compose на сервере.
2. Скопируй шаблон переменных окружения и заполни реальные значения
   (агент в этот файл не смотрит и коммитить его нельзя):

   ```bash
   cp .env.example .env
   ```

   Обязательные поля:
   - `BOT_TOKEN` — токен, выданный @BotFather.
   - `ADMIN_IDS` — список Telegram ID через запятую, кому разрешён доступ
     к боту (пусто — доступ открыт всем, не рекомендуется).
   - `WG_SERVER_ENDPOINT` — публичный `host:port`, к которому будут
     подключаться клиенты (должен резолвиться в твой сервер; порт должен
     совпадать с `WG_SERVER_PORT`).
   - `WG_EGRESS_IFACE` — выходной сетевой интерфейс хоста, который
     используется в правиле MASQUERADE (обычно `eth0`; проверить можно
     командой `ip -o -4 route show default`).

3. Собрать и запустить:

   ```bash
   docker compose up -d --build
   ```

4. При первом старте контейнер `wireguard` сам генерирует приватный ключ
   сервера в volume `wg_keys` (`/run/wg/server.key`) и кладёт публичный
   ключ в `/run/wg/server.pub`. Бот читает публичный ключ из этого файла
   (read-only mount), если не может получить его запросом к интерфейсу.

## Команды бота

- `/start` — регистрация и справка.
- `/new <имя>` — создать нового peer'а. Бот генерирует пару ключей,
  выделяет следующий свободный IP, добавляет peer'а в `wg0` командой
  `wg set` и присылает готовый `.conf` плюс QR-код для мобильного
  WireGuard-клиента.
- `/list` — список твоих peer'ов с их IP и временем последнего рукопожатия.
- `/delete <имя>` — удалить peer'а из БД и из runtime-сессии WG.

## Почему без `wg0.conf`?

Потому что всё состояние живёт в ядровой WireGuard-сессии + в БД. У этого
подхода есть одно следствие: runtime-сессия не переживает рестарт
контейнера `wireguard` сама по себе. Это решается шагом `reconcile` на
старте бота — он читает всех peer'ов из БД и заново применяет их к `wg0`
через `wg set`.

## Безопасность

- Бот монтирует `/var/run/docker.sock`. По сути это даёт контейнеру
  root-эквивалентный контроль над Docker-демоном хоста. Бот не поднимает
  сетевых сервисов и только опрашивает Telegram наружу, так что
  поверхность атаки ограничена Telegram API и whitelist'ом админов.
- Приватные ключи клиентов никогда не попадают на диск сервера. Они
  создаются в памяти бота, отправляются пользователю в `.conf` и сразу
  отбрасываются.
- В SQLite хранятся только публичный ключ и выданный IP. Даже если БД
  утечёт, ей нельзя выдать себя за peer'а (WireGuard аутентифицирует
  peer'ов по владению приватным ключом).

## Разработка

Структура проекта:

- `wireguard/` — `Dockerfile` и `entrypoint.sh` для контейнера WG.
- `bot/app/` — Python-пакет.
  - `config.py` — настройки из `.env` через `pydantic-settings`.
  - `db/` — модели SQLAlchemy и репозиторий.
  - `services/keys.py` — генерация пары ключей через `wg genkey` / `wg pubkey`.
  - `services/ip_alloc.py` — выделение следующего свободного IP с лимитом
    по количеству peer'ов.
  - `services/wg.py` — обёртка `docker exec` вокруг `wg set` / `wg show dump`
    плюс `reconcile()`.
  - `services/config_builder.py` — сборка клиентского `.conf` и PNG-QR.
  - `handlers/` — aiogram-роутеры `/start`, `/new`, `/list`, `/delete`.
  - `middlewares/` — инъекция сессии БД и whitelist админов.
  - `main.py` — точка входа (инициализация БД, reconcile, polling).

Локальный запуск (вне Docker) против уже работающего контейнера WireGuard:

```bash
cd bot
python -m venv .venv
. .venv/bin/activate   # на Windows: .venv\Scripts\activate
pip install -r requirements.txt
python -m app.main
```
